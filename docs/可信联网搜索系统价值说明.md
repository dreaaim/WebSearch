## 一、为什么需要可信联网搜索

### 1.1 传统搜索引擎的局限性

传统搜索引擎依赖**关键提示词触发**机制，用户必须精准输入特定关键词组合才能获得有价值的结果。这一机制存在天然缺陷：

- **关键词敏感性高**：搜索"苹果"可能得到水果、农业、手机等多种结果，歧义消除完全依赖用户输入质量
- **结果质量不可控**：搜索引擎返回的结果仅基于 PageRank 等算法排序，不对内容可信度做任何判断
- **信息茧房效应**：搜索结果被排名算法进一步强化，权威来源与低质量来源混杂

### 1.2 AI 搜索的致命缺陷：缺乏结果校验

以 RAG（检索增强生成）为代表的 AI 搜索范式，本质上是"**先搜后读再生成**"的流水线。然而这一流程存在一个根本性问题：**AI 搜索没有对搜索结果进行任何事实校验**。

具体表现如下：

| 问题类型 | 具体表现 | 潜在风险 |
|---------|---------|---------|
| **信源可信度缺失** | 不区分权威媒体、自媒体、未知来源 | 引用虚假信息作为事实依据 |
| **内容冲突无感知** | 多条搜索结果出现矛盾时，AI 倾向于相信最后一条或权重最高的一条 | 生成自相矛盾的回答 |
| **时效性判断缺失** | 不评估信息来源的时间有效性 | 对有时效性要求的问题给出过时答案 |
| **事实提取粗糙** | 直接将搜索片段拼接给 LLM，未做结构化事实提取 | 断章取义、误解原意 |
| **幻觉放大效应** | LLM 对不准确的事实进行"顺理成章"的推理和扩展 | 错误逐级放大，最终输出严重偏离事实 |

### 1.3 可信联网搜索的核心理念

可信联网搜索系统（Trusted Web Search）旨在解决上述两个核心问题：

> **在联网搜索过程中，引入完整的事实校验机制，确保最终输出的内容不仅相关，而且可信。**

这一理念包含三层含义：

1. **可信**（Trustworthy）：信息来源经过验证，事实陈述经过多源碰撞检测
2. **可溯**（Traceable）：每一条输出结论都能追溯到具体的权威来源
3. **可校验**（Verifiable）：输出结论的可信度可以被量化评估

## 二、可信联网搜索能做什么

### 2.1 覆盖所有需要联网搜索的任务场景

可信联网搜索系统可服务于一切需要获取外部信息的任务：

| 任务类型 | 具体场景 | 输出价值 |
|---------|---------|---------|
| **热点搜索** | 实时事件追踪、突发事件监测、舆情分析 | 返回经过冲突检测的热点事实清单，而非混杂的网页链接 |
| **信息收集** | 竞品分析、行业调研、人物背景调查 | 聚合多源信息，呈现不同角度的事实，支持交叉验证 |
| **针对性分析** | 商业决策支持、技术方案选型、投资分析 | 输出带有可信度评分的事实结论，支持决策参考 |
| **知识问答** | 百科式问答、数据查询、定义解释 | 提供有来源支撑的准确答案，而非 LLM 幻觉 |
| **内容创作辅助** | 事实核查、文章配图依据、内容合规检查 | 提供可查证的事实底料，降低内容风险 |

### 2.2 输出更相关、更可信的内容

与传统搜索相比，可信联网搜索系统的输出质量有本质提升：

**传统搜索输出：**
```
查询："某科技公司竞品最新动态"

返回：["链接1: xxx新闻", "链接2: xxx论坛", "链接3: xxx自媒体", ...]
（用户需自行阅读判断，信息碎片化，无冲突检测）
```

**可信联网搜索输出：**
```
查询："某科技公司竞品最新动态"

返回：
{
  "query": "某科技公司竞品最新动态",
  "trusted_facts": [
    {
      "statement": "竞品X于2026年4月宣布推出新产品线",
      "confidence": 0.92,
      "evidence_sources": ["信源A (权威度:950)", "信源B (权威度:880)"]
    },
    {
      "statement": "竞品Y在华东市场展开促销活动",
      "confidence": 0.78,
      "evidence_sources": ["信源C (权威度:720)"]
    }
  ],
  "collisions": [
    {
      "bucket_id": "bucket_003",
      "collision_coefficient": 0.65,
      "conflicting_facts": [
        "信源D称竞品Z已完成融资",
        "信源E称竞品Z融资谈判仍在进行"
      ],
      "needs_llm_review": true
    }
  ],
  "metadata": {
    "total_sources": 47,
    "trusted_facts_count": 12,
    "collision_count": 3
  }
}
```

### 2.3 可信度量化体系

系统引入了完整的可信度量化体系：

- **信源可信度（Trust Score）**：基于 ELO 动态评分机制，对每个信息来源进行实时可信度打分
- **事实置信度（Confidence Score）**：结合信源可信度和多源交叉验证结果，计算每条事实陈述的置信度
- **碰撞系数（Collision Coefficient）**：量化同一事实桶内不同来源的事实冲突程度

## 三、可信联网搜索的技术实现

### 3.1 系统架构演进

系统经历了三个版本的架构演进：

| 版本 | 核心问题 | 关键改进 |
|------|---------|---------|
| **v1** | LLM 成本高、静态名单无法适应新信源 | 基础搜索 + URL 规则筛选 + Jaccard 碰撞 |
| **v2** | 缺乏结构化事实理解 | QueryRewriter 意图理解 + Embedding 碰撞 + LLM Judge |
| **v3** | 多维度碰撞检测 + LLM 复核机制 | 正交碰撞检测器（SPO/NLI/数值三维度）+ TrustRankLadder ELO 动态评分 |

当前最新的 v3 架构实现了完整的事实校验闭环。

### 3.2 v3 核心处理流程

v3 系统包含以下核心处理阶段：

```
用户查询 → 查询改写 → 多路搜索 → 混合过滤 → 内容获取 → 多因子重排 → LLM精筛 → 事实提取 → 事实桶聚类 → 正交碰撞检测 → 可信事实输出
```

#### 阶段一：查询改写（QueryRewriter）

- **意图分析**：理解用户的真实查询意图
- **查询发散**：生成多个不同角度的查询方向
- **语法增强**：将自然语言转换为搜索引擎友好的查询语法

#### 阶段二：混合过滤引擎（HybridFilterEngine）

结合两种过滤策略对搜索结果进行初筛：

- **Embedding 相似度**：通过语义向量计算结果与查询的相似程度
- **BM25 词频评分**：基于传统信息检索算法的词项权重计算
- **混合评分**：综合两种得分进行排序，平衡语义相关性和词项匹配度

#### 阶段三：多因子重排（MultiFactorReranker）

综合三个维度对内容进行重排：

- **时效性得分**：评估内容的发布时间和内容的新鲜度
- **相关性得分**：基于内容和查询的匹配程度
- **天梯可信度得分**：基于 TrustRankLadder 获取的信源权威性评分

#### 阶段四：LLM 精筛（LLMRefiner）

对重排后的 Top 结果进行 LLM 评估：

- 内容相关性判断
- 内容质量评估
- 事实密度评估

#### 阶段五：事实提取与碰撞检测（FactExtractor + OrthogonalCollisionDetector）

这是系统的核心创新模块：

**事实提取**（FactExtractor）从正文中提取结构化事实：

- **SPO 三元组**：主语-谓语-宾语结构化事实
- **NLI 标签**：自然语言推理标签（蕴含/矛盾/中立）
- **数值信息**：关键数值数据（价格、增长率、百分比等）
- **时间信息**：关键时间节点和期限

**正交碰撞检测**（OrthogonalCollisionDetector）从三个正交维度检测事实冲突：

- **NLI 冲突检测**：检测不同来源的事实陈述是否存在逻辑矛盾
- **SPO 冲突检测**：检测结构化三元组中的主谓宾是否冲突
- **数值/时间冲突检测**：检测关键数值和时间信息是否一致

当碰撞系数超过阈值时，系统触发 LLM 复核机制，由大语言模型对冲突进行最终裁决。

#### 阶段六：可信度天梯（TrustRankLadder）

基于 ELO 评分系统的动态信源可信度管理：

- 新信源初始分数 1000
- 当信源提供的事实被验证为正确时，提升其分数
- 当信源提供的事实被检测为冲突或错误时，降低其分数
- 信源分数实时影响事实的置信度计算

### 3.3 核心模块一览

| 模块 | 功能 | 技术亮点 |
|-----|------|---------|
| **SearchOrchestrator** | 搜索流程编排，协调所有模块 | 统一调度全流程 |
| **QueryRewriter** | 查询意图理解和改写 | 多路发散 + 语法增强 |
| **HybridFilterEngine** | 混合过滤初筛 | Embedding + BM25 双剑合璧 |
| **ContentFetcher** | 网页内容获取 | trafilatura / BeautifulSoup 双引擎 |
| **MultiFactorReranker** | 多因子排序 | 时效性 + 相关性 + 可信度三维加权 |
| **LLMRefiner** | LLM 二次精筛 | Title/Content/URL 综合评估 |
| **FactExtractor** | 结构化事实提取 | SPO/NLI/数值多维提取 |
| **FactBucketCluster** | 相似事实聚类 | Embedding 层次聚类 |
| **OrthogonalCollisionDetector** | 正交碰撞检测 | NLI/SPO/数值三维度冲突检测 |
| **TrustRankLadder** | 信源可信度管理 | ELO 动态评分机制 |

### 3.4 技术栈

系统使用的核心技术组件：

| 组件类型 | 技术选型 | 用途 |
|---------|---------|------|
| 搜索引擎 | SearXNG | 私有化部署的元搜索引擎 |
| Embedding | OpenAI / 本地模型 | 语义向量计算 |
| LLM | OpenAI GPT 系列 | 意图理解、事实提取、冲突裁决 |
| NLP | spaCy / CrossEncoder | SPO 提取、NLI 分析 |
| 聚类 | scipy 层次聚类 | 事实桶聚类 |
| 存储 | JSON / SQLite | 信源注册表、分数持久化 |

## 四、总结

可信联网搜索系统解决了 AI 搜索时代的一个核心矛盾：

> **用户期望 AI 给出准确、可信、有据可查的答案，但 AI 本身缺乏对联网搜索结果进行事实校验的能力。**

本系统通过引入完整的事实处理流水线——从结构化事实提取、到相似事实聚类、再到多维度正交碰撞检测——实现了联网搜索结果的全程可信性校验。结合动态演进的信源可信度评分机制，系统能够持续学习和适应新的信息来源，确保输出内容的可信度始终保持在较高水平。
